# 任务设计方案

## 多任务架构概述

基于 OpenHarmony LiteOS 轻量内核，将系统功能拆分为 5 个独立任务：

```
┌─────────────────────────────────────────────────────┐
│                   LiteOS Scheduler                  │
├───────────────┬──────────────┬──────────────────────┤
│  高优先级     │  中优先级    │  低优先级             │
├───────────────┼──────────────┼──────────────────────┤
│ command_task  │ control_task │ sensor_task          │
│ (指令解析)    │ (设备控制)   │ (环境采集)           │
│ PRIO: 2       │ PRIO: 4      │ PRIO: 6              │
│ Stack: 512    │ Stack: 512   │ Stack: 512           │
├───────────────┼──────────────┼──────────────────────┤
│ alarm_task    │ report_task  │                      │
│ (异常检测)    │ (数据上报)   │                      │
│ PRIO: 3       │ PRIO: 5      │                      │
│ Stack: 384    │ Stack: 768   │                      │
└───────────────┴──────────────┴──────────────────────┘
```

## 任务详细设计

### 1. command_task (指令解析任务)
- **优先级**: 2 (最高)
- **栈空间**: 512 字
- **周期**: 事件驱动 (UART接收中断触发)
- **职责**:
  - 解析云端下发指令帧
  - CRC校验
  - 分发到对应处理器
  - 发送ACK/NAK响应

### 2. alarm_task (异常检测任务)
- **优先级**: 3 (高)
- **栈空间**: 384 字
- **周期**: 1000ms
- **职责**:
  - 读取最新环境数据
  - 逐一检查阈值条件
  - 防抖确认告警
  - 触发蜂鸣器告警
  - 触发设备自动联动
  - 推送告警消息到队列

### 3. control_task (设备控制任务)
- **优先级**: 4 (中)
- **栈空间**: 512 字
- **周期**: 50ms
- **职责**:
  - 接收控制指令队列消息
  - 执行设备开启/关闭
  - 手动按键扫描与消抖
  - 设备运行时间保护
  - LED状态指示

### 4. report_task (数据上报任务)
- **优先级**: 5 (中)
- **栈空间**: 768 字 (协议栈开销大)
- **周期**: 10000ms (10秒上报一次)
- **职责**:
  - 打包环境数据为上报帧
  - 串口发送数据
  - 等待云端ACK
  - 失败重试机制

### 5. sensor_task (环境采集任务)
- **优先级**: 6 (最低)
- **栈空间**: 512 字
- **周期**: 按传感器类型不同 (200ms调度间隔)
  - 温湿度: 2000ms
  - 光照: 1000ms
  - CO2: 3000ms
  - 土壤湿度: 5000ms
- **职责**:
  - I2C/SPI 传感器数据读取
  - 数据滤波处理
  - 信号量保护数据写入
  - 发送数据到消息队列

## 任务同步机制

### 信号量 (Semaphore)
```
g_sem_sensor_data  (1/1) → 传感器数据互斥访问
g_sem_device_ctrl  (1/1) → 设备控制互斥访问
g_sem_uart_tx      (1/1) → 串口发送互斥
```

### 消息队列 (Message Queue)
```
sensor_task ──[sensor_data]──► alarm_task (检测)
                              ├─ report_task (上报)
                              └─ control_task (联动)

cloud ──[cmd]──► command_task ──[device_cmd]──► control_task
                └─[threshold]──► threshold_mgr

alarm_task ──[alarm]──► report_task (上报)
                       └─ control_task (联动)
```

### 优先级设计原则
1. **指令优先**: 确保云端指令10ms内响应
2. **检测次之**: 异常情况快速发现
3. **控制适中**: 设备动作需要一定延时避免抖动
4. **上报适中**: 不影响实时控制
5. **采集最低**: 周期性工作，允许偶尔被抢占

## 系统响应分析

| 场景 | 响应路径 | 预估延时 |
|------|----------|----------|
| 云端指令→设备动作 | command→control→GPIO | ≤10ms |
| 温度超阈值→风扇开启 | alarm→control→GPIO | ≤50ms |
| 数据采集→云端上报 | sensor→report→UART | ≤500ms |
| 按键→水泵启停 | control→GPIO | ≤100ms |

## 栈溢出防护

每个任务栈底放置 Magic Number (0xDEADBEEF)，定期检查:
```c
void stack_guard_check(void) {
    for (int i = 0; i < OS_MAX_TASKS; i++) {
        if (g_tasks[i].stack_base[0] != 0xDEADBEEF) {
            // 栈溢出告警
            sys_set_state(SYS_STATE_FAULT);
        }
    }
}
```
